# Пояснительная записка: Real-time Quiz MVP

## 1. Цель проекта

Цель проекта — разработать работоспособный MVP веб-приложения для проведения квизов в реальном времени. Приложение позволяет организатору создавать квизы, запускать комнату с кодом подключения, показывать вопросы участникам синхронно и получать итоговый лидерборд.

## 2. Выбор средств разработки

Для клиентской части выбран React с Vite. React подходит для интерфейсов с большим количеством состояний: авторизация, конструктор квиза, live-комната, текущий вопрос, ответы и лидерборд. Vite дает быстрый запуск и простую сборку.

Для серверной части выбран Node.js с Express. Express используется для REST API: регистрация, авторизация, создание квизов, создание комнат и получение истории. Для real-time взаимодействия выбран Socket.IO, так как он упрощает работу с WebSocket-соединениями, комнатами и событиями.

Для хранения данных используется SQLite. Это удобный вариант для учебного MVP: база не требует отдельного сервиса, хранится локально в файле `server/database.sqlite` и сохраняет данные после перезапуска сервера.

## 3. Архитектура

Проект разделен на несколько частей:

- `client/` — React-интерфейс пользователя.
- `server/` — Express API и Socket.IO сервер.
- `server/database.sqlite` — локальная база данных.
- `docs/` — материалы для сдачи проекта.

Клиент общается с сервером двумя способами:

- через REST API для постоянных операций;
- через Socket.IO для live-комнат и синхронного прохождения квиза.

В интерфейсе добавлены экран ожидания комнаты, таймер ответа, таблица лидеров, статистика по вопросу, итоговая сводка, экспорт результатов в CSV, история проведенных комнат, дублирование квизов и импорт вопросов из CSV.

## 4. Модель базы данных

В MVP используются таблицы:

- `users` — пользователи, email, пароль в виде hash, роль участника или организатора.
- `quizzes` — квизы, созданные организаторами.
- `questions` — вопросы квиза.
- `answers` — варианты ответов и признаки правильности.
- `rooms` — запущенные комнаты с кодом подключения.
- `roomParticipants` — участники конкретной комнаты и их счет.
- `submissions` — ответы участников на вопросы.

Изображения в вопросах хранятся как `imageUrl`, то есть организатор вставляет ссылку на изображение. Изображение не является отдельным типом ответа: его можно сочетать с одиночным выбором, множественным выбором или текстовым ответом.

## 5. WebSocket-сценарий

Основные события:

- `room:join` — участник подключается к комнате по коду.
- `room:participant-list` — обновление списка участников.
- `quiz:start` — организатор запускает квиз.
- `question:show` — организатор показывает текущий вопрос всем участникам.
- `question:answer` — участник отправляет ответ.
- `question:close` — вопрос закрывается, ответы больше не принимаются.
- `leaderboard:update` — обновляется таблица лидеров.
- `quiz:finish` — квиз завершается, всем показывается финальный лидерборд.

Ответы принимаются только пока вопрос открыт. После закрытия вопроса сервер отклоняет новые ответы, отправляет правильный ответ и статистику по вопросу. После завершения комнаты сохраняется итоговая таблица лидеров, которую можно выгрузить в CSV.

REST API подключения возвращает только публичную информацию о квизе и не раскрывает вопросы или правильные ответы. Организатор подключается к Socket.IO-комнате отдельно и не учитывается среди участников и победителей. Открытый вопрос и срок его завершения сохраняются в SQLite, поэтому участник получает актуальное задание при позднем подключении или после перезапуска сервера.

Для подсчета баллов предусмотрены два режима: стандартный режим с фиксированными баллами за вопрос и режим быстрых ответов, где верный ответ, отправленный раньше, получает больше баллов.

## 6. Этапы разработки

1. Анализ требований и выделение ролей: участник и организатор.
2. Проектирование модели данных.
3. Создание REST API для авторизации, квизов и комнат.
4. Настройка Socket.IO для live-комнат.
5. Разработка React-интерфейса: кабинет, конструктор, запуск комнаты, подключение и лидерборд.
6. Добавление расширений: история комнат, просмотр результатов после завершения, импорт вопросов, копирование квизов и режим быстрых баллов.
7. Тестирование сценария в двух вкладках браузера.

Дополнительно добавлен автоматизированный smoke-тест с организатором и двумя участниками. Он запускается командой `npm run test:smoke` на временной базе данных и не изменяет рабочие данные проекта.

Интерфейс имеет адаптивную верстку для desktop, планшетов и мобильных устройств. На узких экранах навигация, формы, карточки, лидерборд и элементы управления перестраиваются в одну или две колонки без горизонтального переполнения.

## 7. Инструкция запуска

```bash
npm run install:all
npm run dev:server
npm run dev:client
npm run test:smoke
```

Клиент доступен по адресу:

```text
http://127.0.0.1:5173
```

Сервер доступен по адресу:

```text
http://127.0.0.1:4000
```

## 9. Ограничения MVP

Приложение является учебным прототипом. В нем используется локальная SQLite база, простая JWT-авторизация и хранение ссылки на изображение вместо полноценной загрузки файлов. Эти решения позволяют быстро показать основной real-time сценарий без сложной инфраструктуры.
